# Positional Sensitivity of Logit Bias in LLMs

Публичный репозиторий к магистерской диссертации о том, как позиция применения `logit_bias` влияет на наблюдаемый дискурсивный тон ответа большой языковой модели.

## TL;DR

Короткий тезис: один и тот же `logit_bias` работает по-разному в зависимости от того, где он включается внутри генерации. Раннее и срединное вмешательство сильнее меняют маркерный слой ответа, а позднее обычно действует мягче и локальнее.

Практический вывод для DS: управление генерацией нельзя оценивать только как "добавили bias и получили другой текст". Важно смотреть на позицию вмешательства, цену для качества текста и сохранение смысла. В моих экспериментах более сильный сдвиг по дискурсивным маркерам часто сопровождался большей языковой ценой, поэтому `late`-режим оказался более безопасным, но менее сильным, а `early`/`mid` - более чувствительными, но более рискованными.

## Что исследовалось

Работа проверяет, влияет ли момент включения `logit_bias` на поверхностный маркерный слой академического ответа на русском языке.

В качестве прокси дискурсивного тона использовались заранее зафиксированные категории маркеров:

- `logical_structuring`: "во-первых", "однако", "таким образом", "следовательно" и другие структурные связки;
- `hedging_epistemic`: "вероятно", "возможно", "можно предположить" и другие маркеры эпистемической осторожности;
- дополнительные категории для анализа побочных сдвигов: `argument_development`, `intensification_assertiveness`, `anglocentric_formulas`.

Главная идея простая: если текст порождается последовательно, то вмешательство в начале, середине и конце ответа не обязано иметь одинаковый эффект. Начало чаще задает рамку ответа, середина разворачивает аргументацию, а конец оформляет уже сложившуюся траекторию.

## Экспериментальный дизайн

В основном эксперименте сравнивались четыре условия:

| Условие | Что происходит |
|---|---|
| `control` | обычная генерация без `logit_bias` |
| `early` | отрицательный `logit_bias` включается в начале ответа |
| `mid` | отрицательный `logit_bias` включается в середине ответа |
| `late` | отрицательный `logit_bias` включается ближе к концу ответа |

Позиционное вмешательство реализовано через `segment_approximation`: ответ разбивается на сегменты, и `logit_bias` включается только на нужном участке. Это не true token-by-token steering, но воспроизводимый и практически реализуемый способ проверить позиционный эффект через стандартный Chat Completions API.

Основная конфигурация:

| Параметр | Значение |
|---|---|
| Основная модель | `gpt-4.1-mini` |
| Дополнительная проверка | `gpt-4.1-nano` |
| Температура | `0.3` |
| `top_p` | `1.0` |
| Максимальная длина | `180` токенов |
| Повторов на условие | `4` |
| Промптов | `12` |
| `bias_value` | `-8` |
| Основные bias-категории | `logical_structuring`, `hedging_epistemic` |
| Сегменты | `early_end = 0.25`, `mid = 0.40-0.60`, `late_start = 0.75` |

## Метрики

В эксперименте намеренно разведены три разных измерения:

| Метрика | Что показывает |
|---|---|
| `total_marker_score` | нормализованная представленность дискурсивных маркеров |
| `delta_p0` | сдвиг маркерного слоя относительно `control` для того же промпта |
| `cosine_similarity` | близость к контрольному ответу по токеновым счетчикам |
| `perplexity` | proxy языковой цены через биграммную модель, обученную на `control` |

Важно: `delta_p0`, `cosine_similarity` и `perplexity` читаются вместе. Сильный маркерный сдвиг сам по себе не означает хороший steering, если текст при этом теряет смысловую близость или становится менее стабильным.

## Основные результаты

### Основной профиль: `gpt-4.1-mini`

| Условие | Runs | Marker score | `delta_p0` | Similarity | Proxy perplexity |
|---|---:|---:|---:|---:|---:|
| `control` | 48 | 0.022198 | 0.000000 | 1.000000 | 430.657701 |
| `early` | 48 | 0.018377 | -0.003821 | 0.635795 | 708.251565 |
| `mid` | 48 | 0.018029 | -0.004170 | 0.683362 | 642.660427 |
| `late` | 48 | 0.020458 | -0.001741 | 0.707965 | 615.383812 |

Что видно:

- все интервенционные режимы снижают средний маркерный слой относительно `control`;
- `early` и `mid` дают более сильный сдвиг, чем `late`;
- `late` лучше сохраняет similarity и имеет меньшую языковую цену;
- `early` дает сильный эффект, но дороже обходится текстовой стабильности;
- самый заметный вклад в общий сдвиг дает категория `logical_structuring`.

### Проверка на `gpt-4.1-nano`

| Условие | Runs | Marker score | `delta_p0` | Similarity | Proxy perplexity |
|---|---:|---:|---:|---:|---:|
| `control` | 48 | 0.017196 | 0.000000 | 1.000000 | 427.378569 |
| `early` | 48 | 0.016681 | -0.000515 | 0.614393 | 705.416285 |
| `mid` | 48 | 0.015500 | -0.001696 | 0.674498 | 639.396919 |
| `late` | 48 | 0.016551 | -0.000645 | 0.701033 | 608.995428 |

На второй модели общий контур сохраняется: позиция вмешательства влияет на результат, `late` выглядит более щадящим, а ранние и срединные режимы чаще сильнее затрагивают структуру ответа.

## Главный вывод магистерской

Позиция применения `logit_bias` является самостоятельной исследовательской переменной. В выбранной конфигурации позднее вмешательство обычно слабее и локальнее, а раннее и срединное вмешательство сильнее меняют дискурсивный маркерный слой. При этом сила эффекта связана с ценой: чем заметнее сдвиг маркеров, тем выше риск ухудшения текстовой стабильности.

Это полезно для задач production-LLM и DS-экспериментов с управляемой генерацией:

- не стоит считать `logit_bias` только глобальным переключателем стиля;
- позиция применения может быть важна не меньше, чем само значение bias;
- ранний steering может менять риторическую траекторию, но чаще стоит дороже;
- поздний steering может быть безопаснее, если нужно локально подправить завершение;
- оценка должна включать не только силу сдвига, но и сохранение смысла, и языковую цену.

## Как запустить

Требования:

- Python 3.11+
- `OPENAI_API_KEY`
- зависимости из `experiment/requirements.txt`

Установка:

```bash
python3 -m pip install -r experiment/requirements.txt
```

Основной профиль:

```bash
export OPENAI_API_KEY=...
python3 experiment/run_experiment.py --config experiment/config_vkr_max_mini.yaml
```

Проверка на второй модели:

```bash
export OPENAI_API_KEY=...
python3 experiment/run_experiment.py --config experiment/config_vkr_max_nano.yaml
```

Сравнение профилей:

```bash
python3 experiment/compare_profiles.py \
  experiment/outputs_vkr_fast \
  experiment/outputs_vkr_plus \
  experiment/outputs_vkr_max_mini \
  experiment/outputs_vkr_max_nano \
  --out experiment/profile_comparison.csv
```

## Ограничения

- Позиционное управление реализовано через `segment_approximation`, а не через true token-by-token decoding.
- `delta_p0` - proxy по частоте маркеров.
- `perplexity` - локальный proxy на биграммной модели по `control`.
- Основной материал - русскоязычные академические мини-ответы, поэтому перенос на другие жанры требует отдельной проверки.
